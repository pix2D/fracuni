# Playwright for PDF generation and browser tests

Invoice PDFs are generated by rendering HTML templates through headless Chromium via Playwright. This lets us reuse the same HTML/CSS used for browser preview as the PDF source — one template, two outputs. Playwright also serves as the browser provider for Vitest's browser-mode tests, so one stack covers both production rendering and in-browser test assertions.

**Browser provisioning.** The browser is fetched by an explicit `playwright install chromium` in the project's postinstall script:

```json
"scripts": { "postinstall": "playwright install chromium" }
```

Browsers land in `PLAYWRIGHT_BROWSERS_PATH` (a shared/persistent cache in the dev sandbox; `/ms-playwright` in the production image), so the postinstall is a fast no-op once cached. The production Docker image uses the official `mcr.microsoft.com/playwright` base (pinned to the lockfile's Playwright version), which already carries the matching Chromium and its OS libraries.

**Testing.** `vitest.config.ts` defines two projects: a **node** project (data layer, API routes, the PDF pipeline — they need `better-sqlite3` + the filesystem and a real Chromium launch) and a **browser** project (`*.browser.test.ts`, run in headless Chromium via the Playwright provider) for DOM/computed-style/layout assertions on the template.

**Architecture for testability.** The Playwright call lives in a thin `pdf-renderer.ts` (`HTML → PDF buffer`); the rest of the pipeline takes the renderer as an injected dependency, so most tests run with a fake renderer (fast, deterministic) while a small integration test exercises real Chromium end-to-end.

**Considered:** `@react-pdf/renderer` (limited layout primitives), `pdfkit` (manual positioning), Typst (less mainstream). All would require maintaining a separate template system from the web preview.
